# Web OSINT Source Workbench Implementation Plan - 2026-06-19

## Direction

The next build should make the Web OSINT platform usable as a human-led evidence workbench. The priority is not another model integration. The priority is to give captured evidence a durable review layer where the user can select source-linked evidence, annotate it, create proposed facts, link entities, and later promote reviewed material into claims and publication drafts.

The system must preserve this separation:

```text
captured evidence
-> derived observations
-> curated assertions
-> published claims
```

Everything generated by extractors, OCR, VLMs, entity models, relation models, or LLMs starts as a derived observation or suggestion. Nothing becomes a curated assertion without a source anchor and review state.

This plan is now interpreted through the broader 21-job architecture review:

```text
docs/DERIVED_ARCHITECTURE_IMPLEMENTATION_PLAN.md
```

That document folds the research-agent outputs into the larger architecture: source-native adapters, immutable capture bundles, EvidenceDocument revisions, observations, curated assertions, retrieval/model evaluation gates, metrics maturity, and publication snapshots.

First-cut review foundation already implemented:

Main remaining gap:

- The Research UI has a first durable review landing zone, but it is not yet the full workbench. Entity links, claim records, normalized-content corrections, review-state transitions, source/artifact navigation, review-task Inbox rows, and publication draft objects still need first-class persistence and UI flows.

## Implementation Principle

For v1, use append-only review events as the write path.

Research UI actions should produce review events such as:

```text
evidence.selected
annotation.created
annotation.updated
proposed_fact.created
proposed_fact.reviewed
entity_link.created
entity_link.reviewed
claim.created
claim.reviewed
publication_draft.created
```

These events can be materialized into ClickHouse tables for UI queries and analytics. They can later be bridged into Pebble exact lookup or compacted state topics where needed.

## P0 Implementation Slices

### Slice 1 - Durable Review Foundation

Goal: every analyst action in the Research UI becomes durable.

Status: first cut implemented. Keep this section as the intended target shape and the migration checklist from direct ClickHouse/JSONL persistence to the preferred Redpanda review-event materializer.

Deliverables:

- Add a `research_review_event` schema.
- Add a `research.review.events.v1` Redpanda topic.
- Add ClickHouse tables:
  - `research_review_events`
  - `evidence_selections`
  - `annotations`
  - `proposed_facts`
  - `entity_links`
  - `claim_records`
- Add a review materializer worker or extend the Research UI with a strictly append-only internal writer for the first cut.
- Add safe Research UI POST endpoints:
  - `POST /api/review/events`
  - `POST /api/evidence/selections`
  - `POST /api/annotations`
  - `POST /api/proposed-facts`
  - `POST /api/entity-links`
- Add `GET /api/source/review?id=<evidence_id>` to hydrate review state for a source.

Preferred production shape:

```text
Research UI POST
-> research.review.events.v1
-> review materializer
-> ClickHouse review/proposed-fact/entity tables
-> Research UI hydrate endpoints
```

Acceptable first implementation if Redpanda write plumbing slows the slice:

```text
Research UI POST
-> append-only JSONL under /mnt/data/x-research/review/
-> ClickHouse INSERT
```

If the direct path is used, keep the event shape identical to the Redpanda topic so it can be migrated without UI changes.

Validation:

- Create an annotation on the Datalab source.
- Create an evidence selection from an EvidenceDocument block.
- Create one proposed benchmark fact.
- Reload browser and confirm all review objects persist.
- Confirm all new rows are queryable from ClickHouse.
- Confirm no captured artifact is modified.

## P1 Implementation Slices

P1 workers must write suggestions into the review layer. They must not bypass human review or mutate captured evidence.

### Deterministic Entity Extractors

Build deterministic extractors for:

- URLs
- X handles
- GitHub repos
- Hugging Face model IDs
- arXiv IDs
- DOIs
- semantic versions
- model sizes
- hardware/SKU strings
- metric values and units

Output should create `entity_mention` suggestions, not accepted canonical entities.

### GLiNER Entity Worker

Add a GLiNER-based worker only after the review/event model is in place. It should produce suggested entity mentions with source spans and review status.

### Structured Extraction Worker

Add a schema-specific proposed-fact worker after the proposed-fact tables and UI exist. Begin with deterministic/table-based extraction, then NuExtract/small-LLM candidates.

### Factuality Worker

Add DeBERTa/MiniCheck/HHEM only after claims/proposed facts have stable IDs. It should score claim-passage pairs and never decide global truth.

## P3 Retrieval And Model Evaluation

The research-agent outputs were consistent that retrieval/model upgrades need private Web OSINT evaluation before adoption.

Add evaluation sets before enabling neural sparse, late-interaction, or visual reranking lanes:

- exact-name and ID recall: handles, repos, model names, versions, benchmark names, arXiv IDs, hardware SKUs, and URLs
- filtered hybrid search: account, source type, date, project, media present, benchmark topic
- evidence-support retrieval: find the source block/table/region that supports a known claim
- visual evidence retrieval: screenshots, chart/table images, benchmark figures, UI captures, video frames
- review productivity: accepted evidence/facts per review minute and citation correctness

Candidate lanes remain experiments until they beat the baseline on these evals:

- neural sparse candidate retrieval
- late-interaction text rerank over small candidate sets
- visual document retrieval/rerank
- model-neutral listwise rerank interface

## P4 Metrics Maturity

Use the per-tool research jobs to mature the separate metrics dashboard, not the Research UI:

- Redpanda/Connect: topic throughput, consumer lag, DLQ, pipeline status, partition health, and semantic app metrics.
- Pebble: app-owned `DB.Metrics()`, read/write smoke, key counts by namespace, materializer lag, freshness, and compaction health.
- Typesense: health/debug/metrics, query latency, no-hit rate, facet/filter usage, schema drift, and per-document import failures.
- Qdrant: readiness, named-vector coverage, branch freshness, upsert failures, optimizer backlog, telemetry, and snapshot state.
- ClickHouse: ingest rows/bytes, freshness lag, query latency, parts/merges/mutations, active queries, errors, and report-readiness marts.

## P5 Publication And Reuse

After the workbench supports curated evidence, claims, and comparison rows, add frozen publication snapshots:

- narrative changes
- entity-page changes
- comparison/benchmark table changes
- claim additions/removals
- citation/source changes
- unsupported-claim checks
- missing-anchor checks
- unresolved-contradiction checks
- stale/broken-source checks
- unreviewed-AI-suggestion checks

The public research site should read approved publication snapshots, not live mutable internal records.
